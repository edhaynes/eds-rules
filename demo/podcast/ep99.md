## 0:00 — The hook
[CARD: "67.5  vs  42.5"]

Ninth on the whole board against ninety-second. One makes your build the same today as
it was last month. The other is about how you arrange your files. One is reproducibility;
the other is taste.

## 0:20 — The good one
[CARD: "Rule 84 · Pin it and lock it · 67.5"]

"Pin versions and commit the lockfile."

Sixty-seven and a half — five plain words that punch way above their weight. Without a
pin and a committed lockfile, your dependencies float. The build that passed this
morning pulls a different library version this afternoon and breaks, and you've got no
idea why because nothing *you* wrote changed. Pinning freezes the world. The lockfile
in git means anyone who clones gets the exact same versions you did — same build, same
machine, next year. That's the floor under reproducibility, and it scores high because
without it nothing else you test is trustworthy.

## 1:10 — The bad one
[CARD: "Rule 33 · One non-trivial class per file · 42.5"]

Now forty-two and a half. "One non-trivial class per file; small helpers and DTOs may
share."

This is fine advice. It keeps files navigable, keeps the diff readable, keeps you from
that two-thousand-line monster with nine classes in it. But it's a style preference, and
a soft one — note the "non-trivial" and the carve-out for helpers. Break it and your
code is slightly harder to navigate. That's it. It scores in the low forties because the
cost of ignoring it is mild and the rule itself is more convention than law.

## 2:00 — The lesson
[CARD: "reproducible vs readable"]

That's the split. Rule eighty-four makes your build *reproducible* — break it and you
can't trust anything, including your own tests. Rule thirty-three makes your code
*readable* — break it and a teammate sighs. Both worth following; the grade just knows
that a broken build is a crisis and a crowded file is a preference. Reproducibility is
load-bearing. Layout is comfort.
