## 0:00 — The hook
[CARD: "65.5  vs  44.0"]

Two rules out of the same git workflow today. One makes you read the config diff before
you commit — the place a leaked key hides best. The other tells you how to type a `git
push` when there's a tag involved. One stops a credential from going public; the other
stops you tripping over your own tags. Same tool, very different stakes.

## 0:20 — The good one
[CARD: "Rule 56 · Inspect config-file diffs · 65.5"]

"Before any commit, show and read the staged diff of every config-shaped file —
`.env`, yaml, json, anything under infra or deploy — even files that were already in the
repo. A leaked key hides best inside an ordinary config commit by another agent."

Sixty-five and a half. This is the rule that catches the leak nobody meant to make. You
don't usually paste an `sk-` key into your own code — you do it by adding a config file
you half-glanced at, or by committing a yaml another tool generated. So the discipline
is: before the commit, the staged diff of every config-shaped file goes up on screen and
gets *read*. Not "I changed config, looks fine." Read the lines. And read them even when
the file was already tracked — trust no prior commit blind. It scores high because the
failure it prevents is the loudest one there is: a live credential in public history.

## 1:10 — The bad one
[CARD: "Rule 68 · Push tags by name · 44.0"]

Now forty-four. "Push tags by name — `git push origin v1.2.3` — never `git push --tags`
reflexively, so an accidentally-rewritten local tag can't clobber the remote."

This is real advice and I follow it. `--tags` shoves every local tag at once, and if one
of your local tags drifted from what's on the remote, that push quietly overwrites a
released tag — and a tag people pinned to just moved under them. Naming the one tag you
mean keeps that from happening. Good habit. But look at the blast radius: it's a tag, on
a release, in a repo that bothers with tags at all, on the rare day your local tags and
the remote disagree. Narrow, conditional, low-frequency. Mid-forties is exactly right.

## 2:00 — The lesson
[CARD: "every commit vs the odd tag push"]

That's the split. Rule fifty-six fires on *every commit you ever make* and guards against
the worst outcome on the board — a secret in public history. Rule sixty-eight fires on
the rare tagged push and guards against a self-inflicted tag clobber you can usually undo.
Both are git hygiene; the grade ranks by how often the rule earns its keep and how bad the
miss is. Read your config diffs every single time. Type your tag pushes carefully the few
times it comes up. The rubric just tells you which one to never, ever skip.
